# Deploying Anchor (Render)

_Decision (INF-5, 2026-07-13): production runs on Render as a managed PaaS._
_Rationale: the `render.yaml` blueprint maps 1:1 onto the docker-compose
topology already verified locally (nginx web entrypoint, private API,
worker, managed Postgres/Redis), with TLS, daily DB backups, and deploy-
on-push included and no servers to patch._

## Topology

| Compose service | Render resource | Exposure |
|---|---|---|
| `frontend` (nginx + SPA) | `anchor-frontend` — Docker **web service** | Public, TLS via Render |
| `backend` (FastAPI) | `anchor-api` — Docker **private service** | Internal only; reached through the nginx `/v1` proxy |
| `worker` (scheduler) | `anchor-worker` — Docker **background worker** | None |
| `migrate` (one-shot) | `preDeployCommand: uv run alembic upgrade head` on `anchor-api` | Runs once per deploy, before new instances start |
| `db` | `anchor-db` — managed Postgres | Internal only |
| `redis` | `anchor-redis` — managed Key Value, `noeviction` | Internal only |

The API being a *private service* means the only internet-facing surface is
nginx with its CSP/security headers; `/v1/docs` and the API are unreachable
except through the proxy.

## First deploy

1. Push to GitHub (already done — `main` is the deploy branch).
2. Render dashboard → **New → Blueprint** → select this repo. Render reads
   `render.yaml` and creates all six resources.
3. Fill the secrets marked `sync: false` when prompted:
   - `ANTHROPIC_API_KEY` — from the Anthropic console.
   - `CORS_ORIGINS` — the public frontend URL (e.g. `https://anchor-frontend.onrender.com`,
     later the custom domain). Same-origin requests through the proxy don't
     need CORS, but keep this accurate for direct-origin edge cases.
   - `PUBLIC_APP_URL` — same public URL; used in password-reset emails.
4. `JWT_SECRET`, `MAGIC_LINK_SECRET`, and `FIELD_ENCRYPTION_KEY` are
   **generated by Render** on the API service and copied to the worker via
   `fromService` — do not set them by hand, and never rotate
   `FIELD_ENCRYPTION_KEY` casually (it decrypts stored coach messages, RSD
   triggers, and mood notes; see Key rotation below).
5. The app **refuses to boot** with `APP_ENV=production` and weak secrets,
   a missing encryption key, or `APP_DEBUG=true` (`app/core/config.py`) —
   a failed first boot is most likely a missing secret, check the logs.
6. Email: SMTP vars are unset by default, so password-reset mails log to
   console. Set `SMTP_HOST/PORT/USERNAME/PASSWORD` + `EMAIL_FROM` on
   `anchor-api` when a provider is chosen (any SMTP provider works).

## Verifying a deploy

- `https://<frontend-url>/v1/health` → `{"status":"ok","db":"ok","redis":"ok"}`.
- Register a throwaway account, run one coach exchange, delete the account
  (Settings → Delete) — exercises DB, Redis, AI, and the deletion path.
- Logs: both services emit JSON logs with `request_id`; the deploy fails
  fast if migrations fail (preDeploy runs before instances flip).

## Backups & restore (DATA-3)

- **Backups**: the paid Postgres plan takes automatic **daily snapshots**
  (7-day retention on basic plans). Point-in-time recovery is available on
  higher tiers if/when needed.
- **Restore drill** (do this once before launch, then quarterly):
  1. Render dashboard → `anchor-db` → Recovery → restore snapshot to a
     **new** instance (never in place).
  2. Point a review copy of `anchor-api` at the restored instance's
     connection string and check `/v1/health` + a data spot-check.
  3. To promote: update `DATABASE_URL` on `anchor-api` + `anchor-worker`.
- **Manual export** (off-platform copy, e.g. before risky migrations):
  `pg_dump "$(render db url anchor-db)" | gzip > anchor-$(date +%F).sql.gz`
  Store in a private bucket; these dumps contain encrypted-at-rest fields
  but also plaintext user data — treat them as sensitive.
- **Key Value data** (rate limits, idempotency replays, token denylist) is
  operational, not user data: acceptable to lose on failover. Losing the
  denylist un-revokes tokens for up to their 7-day lifetime — after a Redis
  wipe, rotate `JWT_SECRET` if any revocation mattered.

## Data retention (working policy — formalize before public launch)

- User content lives until the user deletes it; account deletion cascades
  all content immediately (immediate mode) or after a 30-day grace period
  (scheduled mode, executed by `anchor-worker`).
- DB snapshots age out on the provider's schedule (7 days) — a deleted
  account is therefore fully gone from backups within a week.
- `rsd_logs` are stored pseudonymously (never linked to a user id).
- Privacy Policy and Terms of Service **drafts** live in `docs/legal/`
  (`PRIVACY_POLICY.md`, `TERMS_OF_SERVICE.md`) — grounded in the app's real
  data practices but **must be finalized by counsel** and published before
  public launch.

## Key rotation

- `JWT_SECRET`: rotate freely — it logs everyone out, nothing else.
- `FIELD_ENCRYPTION_KEY`: rotation requires re-encrypting
  `coaching_messages.content`, `rsd_logs.trigger_text_enc`,
  `mood_checkins.note_enc`, and `ai_messages` feedback notes with the new
  key (write a one-off migration mirroring `0012`). Never change the value
  without that migration — stored text becomes unreadable.

## Custom domain / HSTS

Add the domain on `anchor-frontend` (Render provisions the cert). The API
already sends `Strict-Transport-Security` when `APP_ENV=production`;
consider preloading the domain once it's stable.

## What stays local

`docker-compose.yml` remains the dev environment (Ollama AI engine, seeded
test account). Nothing in it is used in production.
